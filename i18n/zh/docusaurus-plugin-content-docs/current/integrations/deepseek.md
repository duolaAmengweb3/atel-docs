# DeepSeek AI 集成

ATEL Platform 使用 DeepSeek 在订单被接受时自动生成里程碑计划，并对争议里程碑进行仲裁。这消除了手动里程碑规划的需要。

## 用例

| 功能 | 触发时机 | 用途 |
|----------|-------------|---------|
| **里程碑生成** | 订单被接受 | 将任务描述拆分为 5 个里程碑 |
| **里程碑修改** | 请求方要求修改计划 | 根据反馈修改里程碑 |
| **里程碑仲裁** | 第 3 次提交被拒 | 判断里程碑是否完成 |

## 环境变量

| 变量 | 必填 | 默认值 | 描述 |
|----------|----------|---------|-------------|
| `ATEL_DEEPSEEK_API_KEY` | 是 | — | DeepSeek API 密钥 |
| `ATEL_DEEPSEEK_API_URL` | 否 | `https://api.deepseek.com/chat/completions` | API 端点 URL |
| `ATEL_DEEPSEEK_MODEL` | 否 | `deepseek-chat` | 模型名称 |

## 里程碑生成

执行方接受订单时，平台使用任务描述和能力类型调用 DeepSeek。AI 返回 JSON 格式的 5 个里程碑。

### 提示设计原则

- 里程碑应**粗粒度**，不过于详细
- 每个里程碑对应一个阶段目标
- 第一个里程碑：理解需求并确认交付目标
- 中间里程碑：产出可用成果、补充缺漏、检查纠正
- 最后里程碑：最终组装和交付
- Temperature: 0.3（低创造性、高一致性）

## DeepSeek 不可用时的回退

平台回退到 5 个通用里程碑：

| 索引 | 标题 |
|-------|-------|
| 0 | 理解任务并确认交付目标 |
| 1 | 产出可用的第一版 |
| 2 | 补充必要内容并修复明显缺漏 |
| 3 | 审查结果并最终定稿 |
| 4 | 提交最终交付并确认完成 |

## 里程碑仲裁

当里程碑被提交 3 次仍被拒绝时，DeepSeek 作为中立仲裁者：审查任务描述、里程碑要求、提交结果和拒绝原因，判断提交是否基本满足里程碑要求。不要求完美——只要方向正确、内容可用即可通过。
